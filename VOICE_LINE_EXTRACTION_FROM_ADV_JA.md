# ADV 由来のボイスライン抽出メモ

この文書では、`abdata/adv/scenario` bundle から `hsa_*` の音声クリップと、それに対応するセリフ本文を実用的に抽出する方法をまとめます。

## 目的

目的は、ADV の分岐ロジック全体を完全に解釈することではありません。
必要なのは、実用的な CSV 形式で次の対応を得ることです。

- `clip`
- `text`
- 必要に応じて bundle 名や scenario block 名などの出典情報

これは、UI 表示、検索、ボイスライン補完に使うことを想定しています。

## なぜこの方法で取れるのか

現在までの調査では、多くの ADV scenario block で、音声再生と本文表示が近い command 列に並んでいます。

よくある並びは次のとおりです。

- `17` = 音声再生
- `16` = セリフ表示

多くの場合、対応する本文は
- `17` の直後
- あるいは少数の軽い補助 command を挟んだ後

に現れます。

そのため、ADV VM 全体を完全に再現しなくても、高精度の抽出器を作れます。

## 推奨する抽出戦略

### Step 1: 1 性格分の scenario bundle をすべて読む

対象ディレクトリの形は次のとおりです。

```text
abdata/adv/scenario/cXX/<bundle-set>/
```

例:
- `abdata/adv/scenario/c02/30`
- `abdata/adv/scenario/c13/50`

### Step 2: すべての `ADV.ScenarioData` MonoBehaviour を読む

各 `ScenarioData` について、次を読みます。

- `m_Name`
- `list.Array`
- 各 item の `_command` と `_args`

### Step 3: 音声 command を検出する

`_command == 17` を見つけたら:
- `_args` を読む
- 最後の文字列引数が、通常は clip 名 (`hsa_*`) になる
- その clip 名が対象性格 prefix (`hsa_02_`, `hsa_13_` など) で始まる場合だけ処理を続ける

### Step 4: 対応する本文を探す

そこから少し先を見て、たとえば 6 から 8 item 先まで探索します。

別の強い境界に当たる前に `_command == 16` が見つかったら、その本文を採用します。

実用的な停止条件:
- 次の `17` に到達したら止める
- 新しい label block (`12`) が始まったら止める
- choice block (`24`) が始まったら止める

実用的な許容条件:
- `17` と `16` の間に軽い presentation / control command が入っていても許容する

この方法は、少なくとも `022` / `024` 系でかなり有効です。

## ペアリング時に通常は解釈不要な command

次の command は、音声 command と本文の間や、その近くの presentation ロジックとして出ることがあります。

- `18`
- `19`
- `98`
- `104`
- `120`

最初の実装では意味まで解釈する必要はありません。
単に「本文探索中は読み飛ばしてよい command」として扱えば十分です。

## 推奨 CSV 形式

抽出用の一次 CSV には、次の列があると便利です。

- `clip`
- `text`
- `bundle`
- `scenario_name`
- `item_index`
- `command_window`
- `source_bundle_path`

これにより、後から抽出結果をデバッグしやすくなります。

## 推奨する重複整理後 CSV 形式

アプリ側で使う二段階目の CSV では、次の列が扱いやすいです。

- `clip`
- `text`
- `candidate_rows`
- `unique_texts`
- `sample_bundle`
- `sample_scenario`

これにより、1 clip 1 行にしつつ、曖昧さの情報も保持できます。

## 現在の試作結果

この方法を `c02` に適用した private 側の試作では、次の結果になりました。

- 生行数: `1208`
- unique clip 数: `861`

精度はかなり高く、ボイスライン補完用途には十分に有望でした。

## 既知の限界

この方法は意図的にヒューリスティックです。
ADV の制御フロー全体は解釈しません。

既知の限界:
- 同じ clip が複数回出る場合がある
- 1 clip に複数候補の本文が付く場合がある
- 一部の分岐では、より深い label 解決が必要になる
- 一部の本文は `list/h/sound/voice` にしか存在しない可能性がある
- 本文が近接していない場合は拾い損ねることがある

## 実運用での推奨

実装コストと網羅率のバランスを考えると、次の 2 層構成が良いです。

1. `list/h/sound/voice` 側の既存本文を優先して読む
2. 足りない部分を ADV 側の `adv/scenario` から `17 -> 近傍の 16` ルールで補完する

この構成なら、比較的小さい実装でかなり高いカバー率を得られます。

## 今後の改善案

より高い精度が必要になったら、次の順で拡張するのが良いです。

1. `17` と `16` の間に許容する skip command を増やす
2. 1 clip に複数候補本文を保持できるようにする
3. 近傍の label と単純な分岐を解決する
4. それでも必要な場合に限って、ADV command VM の本格的な解釈に進む
